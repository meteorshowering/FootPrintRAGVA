# 数据分析建议与系统评估框架

本文档给出两类内容：**一、可基于现有实验 JSON 开展的数据分析**；**二、多轮编排式 RAG 系统应如何评估**（指标、基线、实验设计）。可直接用于论文的「实验设计 / 评估」章节或作为后续实验与脚本开发的依据。

---

## 一、可做的数据分析（基于现有 JSON）

实验日志 `experiment_results_20260222_164140.json` 中每条证据包含：检索得分 `score`、评估 `branch_action`（GROW/KEEP/PRUNE）、`relevance`/`credibility`（1–10）、`paperid`、`suggested_keywords`；每轮有 `query_intent`、`total_results`、`duplicate_results`。以下分析均可仅用该文件完成。

### 1. 检索得分与评估结果的一致性

- **问题**：向量检索的相似度分数是否与「高价值」一致？
- **做法**：对每条有 `evaluation` 的证据，取 `retrieval_result.score` 与 `evaluation.scores.relevance`（及 `credibility`）做相关分析；按 `branch_action` 分组计算检索得分的均值/中位数。
- **预期**：若系统校准良好，GROW 的检索得分与 relevance 应高于 PRUNE；若检索得分与 relevance 相关性弱，说明编排与评估在修正「纯检索排序」。
- **可输出**：Pearson/Spearman 相关系数；按 branch_action 的 score / relevance / credibility 箱线图或表格。

### 2. 轮次层面的收敛与去重

- **问题**：多轮是否在「收敛」、重复是否随轮次增加？
- **做法**：按 `round_number` 汇总每轮的 `total_results`、`duplicate_results`，计算每轮「新增非重复证据」比例；统计每轮 GROW/KEEP/PRUNE 的数量与占比。
- **可输出**：折线图（轮次 vs. 重复率、轮次 vs. GROW 占比）；表（每轮 total/duplicate/有效条数）。

### 3. 论文级证据分布与「高价值论文」识别

- **问题**：哪些论文被频繁检索且多被标为 GROW？
- **做法**：按 `paperid` 聚合，统计每个 paper 的出现次数、GROW/KEEP/PRUNE 次数；定义简单指标如「GROW 占比」或「GROW 数 − PRUNE 数」排序。
- **可输出**：论文 × 出现次数 × GROW/KEEP/PRUNE 表；TOP-K 高价值 paper 列表（与案例中的 paper_007、paper_017 等可交叉验证）。

### 4. 编排策略与查询演化

- **问题**：`query_intent` 如何随轮次变化？是否从宽到窄？
- **做法**：按轮次列出 `query_intent` 文本；可计算相邻轮次 query 的文本相似度（如 TF-IDF + 余弦）或长度变化。
- **可输出**：各轮 query_intent 列表；可选：相似度矩阵或「首轮 vs. 末轮」对比句。

### 5. 建议关键词与 GROW 的关联

- **问题**：带 `suggested_keywords` 的证据是否更多被标为 GROW？
- **做法**：二分「是否有 suggested_keywords」与 branch_action 做列联表；或统计 GROW 中 suggested_keywords 非空的比例 vs. KEEP/PRUNE。
- **可输出**：列联表；GROW 中关键词频次统计（可词云展示）。

### 6. Relevance 与 Credibility 的分布与区分度

- **问题**：双维度评分是否在不同 branch_action 间有区分度？
- **做法**：按 branch_action 计算 relevance、credibility 的均值与标准差；可做 Kruskal-Wallis 或 Mann-Whitney 检验。
- **可输出**：分组描述统计表；箱线图；若显著，可写「GROW 的 relevance 显著高于 PRUNE（p<0.05）」。

### 7. 汇总统计表（便于写进论文）

在现有统计基础上，可统一输出一张「会话级汇总表」，例如：

| 指标 | 数值 |
|------|------|
| 总轮数 | 7 |
| 总检索调用次数 | … |
| 总证据条数（含未评估） | … |
| 被评估证据条数 | 172 |
| GROW / KEEP / PRUNE 数量与占比 | 69(40.1%) / 53(30.8%) / 50(29.1%) |
| 涉及论文数（唯一 paperid） | … |
| 平均每轮 duplicate_results | … |
| relevance / credibility 均值（全表或按 branch_action） | … |

以上分析均可用 Python 脚本从 JSON 解析后实现（pandas + scipy/statsmodels）；若需要，可再写一份简短的数据分析脚本提纲或示例代码结构。

## 二、系统应如何评估

本系统是多轮、编排驱动的 RAG：**编排器**规划检索策略与 query_intent，**检索**返回文献片段，**评估器**给出 branch_action 与 relevance/credibility。评估应覆盖「检索质量」「编排与演化」「评估一致性」以及「端到端任务效果」；若有条件，可加入人工标注与用户研究。

### 2.1 评估维度与建议指标

| 维度 | 说明 | 建议指标 | 数据来源 |
|------|------|----------|----------|
| **检索质量** | 单次/多轮检索是否命中相关文献 | 检索得分与 relevance 的相关性；按 round 的「至少 1 条 GROW」比例；若有金标，Recall@k、MRR、NDCG | 日志中的 score、evaluation；可选金标文档集 |
| **编排与演化** | 策略与 query 是否合理、是否收敛 | 轮次间 query_intent 多样性/相似度；duplicate_results 随轮次变化；每轮 GROW 占比趋势 | 日志 query_intent、duplicate_results、branch_action |
| **评估一致性** | 评估是否稳定、是否与人类一致 | 同 chunk 在不同上下文的评估是否一致（若有多会话）；与人工标注的 Kappa、相关系数 | 日志 + 抽样人工标注 |
| **覆盖与效率** | 是否用较少轮次/调用覆盖足够多高价值证据 | 达到 N 条 GROW 所需轮数；总检索调用数；唯一高价值 paper 数 | 日志聚合 |
| **端到端任务** | 用户能否基于系统输出完成任务 | 任务完成率、答案事实性/引用准确性（LLM 或人工判）；时间/点击步数（若有交互日志） | 设计任务 + 人工或 LLM 评判 |

### 2.2 基线对比（建议）

- **单轮检索**：仅用根问题做一次语义检索，取 top-k，不做多轮与评估。对比：多轮编排在 Recall、GROW 数量、覆盖论文数上是否更优。
- **固定多轮**：固定 2–3 个 query（如根问题 + 两个预设子问题），无编排器。对比：编排器生成的 query 与轮次是否带来更多 GROW 或更高 relevance。
- **无评估反馈**：多轮检索但不使用评估器的 GROW/KEEP/PRUNE，仅按检索得分排序。对比：引入评估后，用户可见的「高价值证据」比例与人工/任务完成是否更好。

可报告：相对各基线的相对提升（如 GROW 条数 +x%、达到 20 条 GROW 所需轮数 −x%）、以及检索得分与 relevance 的相关性变化。

### 2.3 人工标注与校准（可选但推荐）

- **抽样**：从日志中随机抽 50–100 条证据（跨轮次、跨 branch_action），由 1–2 名领域专家标注：相关性（1–5 或 1–10）、是否建议深挖（是/否/保留），可选「与问题无关」。
- **指标**：与系统 relevance/credibility 的 Spearman 相关；与 branch_action 的混淆矩阵、Kappa；GROW 的精确率/召回率（以人工「建议深挖」为金标）。
- **用途**：论证评估器与人类判断一致；若相关性高，可将「系统 relevance 均值」作为检索质量的代理指标。

### 2.4 用户研究设计（若有资源）

- **任务**：给参与者同一批根问题（如 2–3 个不同领域），要求「在限定时间内梳理方法/结论并写出后续研究建议」；一组用本系统（可多轮+补充问题），一组用基线（如单轮检索或固定多轮）。
- **指标**：任务完成质量（专家打分或 rubric）、所用时间、满意度量表、是否使用了补充问题与 GROW 线索。
- **分析**：组间比较 + 案例引用（可与现有案例研究呼应）。

### 2.5 评估实施优先级建议

1. **先做**：基于现有 JSON 的**数据分析**（第一节 1–7），产出描述统计与图表，写入论文「实验与结果」。
2. **其次**：实现**单轮/固定多轮基线**，在同一批 root_goal 上跑通，对比 GROW 数、覆盖论文数、轮次效率。
3. **若有时间**：做**小规模人工标注**，报告评估器与人的一致性。
4. **若面向应用**：设计**小规模用户研究**，与案例研究结合，说明「从开放问题到结论」的闭环与满意度。
