# ToRAGLENS 后端实现文档 - 论文材料

## 一、系统概述

### 1.1 系统定位
ToRAGLENS 是一个基于多智能体协作的科学知识发现系统，旨在通过多轮迭代的文献检索、评估和推理，构建科研证据树（Research Graph），最终生成高质量的科学回答报告。

### 1.2 核心目标
- **深度探索**：通过多轮迭代，深入挖掘科学问题的多个维度
- **广度扩展**：动态扩展检索范围，发现新的研究方向和关联知识
- **证据链构建**：将碎片化的证据组织成结构化的证据树
- **智能评估**：对检索结果进行质量评估，筛选高价值证据
- **知识综合**：综合所有证据，生成结构化的科学报告

### 1.3 技术架构
系统采用**多智能体协作架构**，基于 AutoGen Core 框架实现，主要包含以下组件：
- **前端接口层**：FastAPI + WebSocket 实时通信
- **智能体系统层**：5个专业智能体协同工作
- **工具执行层**：多种检索工具（语义检索、元数据检索、多模态检索等）
- **数据存储层**：ChromaDB 向量数据库 + 本地 JSON 缓存

## 二、核心智能体系统

### 2.1 OrchestratorAgent（首席科学家/协调者）

#### 2.1.1 职责定位
OrchestratorAgent 是整个系统的"大脑"，负责：
- 接收用户查询，初始化科研证据树
- 基于评估反馈，动态规划下一轮检索策略
- 平衡广度探索（Exploration）和深度挖掘（Exploitation）
- 控制整个工作流的执行流程

#### 2.1.2 核心数据结构

**ResearchGraph（科研证据树）**
```python
class ResearchGraph:
    root_goal: str                    # 用户原始问题
    nodes: Dict[str, GraphNode]        # 所有节点（证据节点 + 根节点）
    action_history: List[ActionTrace]  # 操作历史记录
    evidence_fingerprints: set[str]    # 证据唯一标识集合（用于去重）
```

**GraphNode（图谱节点）**
```python
class GraphNode:
    id: str                           # 节点唯一标识
    type: str                         # 节点类型：ROOT / EVIDENCE
    status: str                       # 节点状态：ACTIVE / PENDING / PRUNED
    content: Dict[str, Any]           # 节点内容（标题、摘要、洞察等）
    metadata: Dict[str, Any]          # 元数据（论文ID、关键词等）
    ParentNode: Optional[str]         # 父节点ID（用于构建树结构）
    children_ids: List[str]           # 子节点ID列表
    hit_count: int                    # 被检索到的次数（重复计数）
    created_at_round: int             # 创建轮次
    search_history: List[NodeSearchRecord]  # 搜索历史记录
    source_tool: str                  # 来源工具
    source_args: Dict[str, Any]       # 来源参数
    source_reason: str                # 生成原因
    evaluation: Optional[ItemEvaluation]  # 评估结果
```

#### 2.1.3 决策策略

**策略类型：**

1. **深度挖掘（DEEPER - Exploitation）**
   - **触发条件**：
     - 评估人员报告某个节点为高价值（GROW）
     - 节点包含可深挖的线索（如 paper_id、关联图表）
   - **操作方式**：
     - 使用 `strategy_metadata_search` 工具
     - 基于父节点的 `paper_id` 或 `keywords` 进行精确检索
     - ParentNode 必须设置为具体节点ID（不能是"0"）
   - **目的**：将孤立的线索扩展为完整的证据链

2. **广度扩展（BROADER - Exploration）**
   - **触发条件**：
     - 当前返回大量重复论文（说明该方向已充分研究）
     - 当前返回论文数量过少（说明策略无效或相关性低）
     - 评估人员建议使用新的专业名词或关键词
   - **操作方式**：
     - 使用 `strategy_semantic_search` 工具
     - 基于新关键词或评估建议进行语义检索
     - ParentNode 可设置为"0"（综合思考）或具体节点ID（基于节点扩展）
   - **目的**：扩展搜索范围，发现新的研究方向和关联知识

**策略生成约束：**
- 每个策略必须满足：相关性评分 ≥ 7 且新颖性评分 ≥ 7
- 参考历史策略，避免重复
- 每轮可生成 1-5 个不同的检索策略（支持并发执行）

### 2.2 ToolExecutorAgent（工具执行者）

#### 2.2.1 职责定位
- 接收 OrchestratorAgent 的批量检索任务
- 并发执行多个检索工具
- 汇总结果并返回给 OrchestratorAgent

#### 2.2.2 实现机制

```python
@message_handler
async def handle_plan(message: OrchestratorPlanBatch, ctx: MessageContext):
    # 1. 定义单个任务执行函数
    async def run_single_task(plan: OrchestratorPlan) -> SingleToolOutput:
        tool_func = ALL_TOOLS_MAP.get(plan.tool_name)
        if tool_func:
            items = await tool_func(**plan.args)
            return SingleToolOutput(original_plan=plan, raw_items=items)
        else:
            return SingleToolOutput(original_plan=plan, raw_items=[], error="Tool not found")
    
    # 2. 并发执行所有任务
    tasks = [run_single_task(plan) for plan in message.plans]
    results = await asyncio.gather(*tasks)
    
    # 3. 汇总结果并返回
    response = ToolOutputBatchMessage(outputs=results)
    await self.publish_message(response, topic_id=TopicId(TOPIC_ORCHESTRATOR))
```

#### 2.2.3 支持的检索工具

系统支持以下检索工具（定义在 `scientific_tools.py`）：

1. **strategy_semantic_search**：语义检索
   - 参数：`query_intent: str`
   - 功能：基于向量相似度的语义检索

2. **strategy_metadata_search**：元数据检索
   - 参数：`paper_id: str`, `keywords: List[str]`, `figure_type: str`
   - 功能：基于论文ID、关键词、图表类型的精确匹配

3. **strategy_multimodal_search**：多模态检索
   - 参数：`query_image_path: str`
   - 功能：基于图片的相似度检索（使用CLIP模型）

4. **strategy_vector_search**：向量检索
   - 参数：`vector: List[float]`, `n_results: int`
   - 功能：直接基于向量进行相似度检索

5. **strategy_evidence_similarity_search**：证据相似度检索
   - 参数：`evidence_id: str`, `n_results: int`
   - 功能：基于现有证据节点的嵌入进行相似证据检索

### 2.3 EvaluatorAgent（评估者）

#### 2.3.1 职责定位
- 对检索到的每条新证据进行单点评估
- 判断证据价值（GROW / KEEP / PRUNE）
- 提取关键洞察和推荐关键词
- 为 OrchestratorAgent 提供决策依据

#### 2.3.2 评估标准

**分支动作（branch_action）：**
- **GROW**：高价值证据。与问题强相关，且包含进一步深挖的线索
- **KEEP**：中等价值。与问题相关，但可能只是背景介绍
- **PRUNE**：低价值。与问题无关，或信息过于模糊/重复/低质

**评估维度（scores）：**
- `relevance`：相关性（0-10）
- `credibility`：可信度（0-10）
- `accuracy`：准确性（0-10）
- `authority`：权威性（0-10）
- `completeness`：完整性（0-10）

**输出信息：**
- `extracted_insight`：提取的关键洞察（结构化描述）
- `suggested_keywords`：推荐的专业名词或关键词（用于扩展搜索）
- `reason`：评估理由

#### 2.3.3 实现机制

```python
@message_handler
async def handle_request(message: EvaluationRequest, ctx: MessageContext):
    # 1. 构建评估Prompt
    prompt_text = "请对以下新检索到的证据进行单点评估（JSON List 格式）：\n"
    for item in message.new_items:
        # 构建证据描述（包含metadata和content）
        item_desc = f"""
        --- Evidence Item ---
        ID: {item.id}
        Source Tool: {item.source_tool}
        Metadata: {json.dumps(meta_preview, ensure_ascii=False)}
        Content: {json.dumps(item.content, ensure_ascii=False)}
        """
        prompt_text += item_desc + "\n"
    
    # 2. 调用LLM进行评估
    response = await self.model_client.create(
        messages=[self.system_message, UserMessage(content=prompt_text)],
        cancellation_token=ctx.cancellation_token
    )
    
    # 3. 解析评估结果
    eval_list_data = json.loads(cleaned_response)
    evaluations = [ItemEvaluation(**d) for d in eval_list_data]
    
    # 4. 返回评估报告
    await self.publish_message(
        EvaluationReportMessage(evaluations=evaluations),
        topic_id=TopicId(TOPIC_ORCHESTRATOR)
    )
```

**批处理机制：**
- OrchestratorAgent 将新证据分批发送（每批5条）
- 避免单次评估过多证据导致LLM上下文过长
- 通过 `pending_eval_batches` 计数器控制批处理完成

## 四、WebSocket实时通信

### 4.1 ConnectionManager

#### 4.1.1 连接管理
```python
class ConnectionManager:
    active_connections: List[WebSocket]  # 活跃连接列表
    
    async def connect(websocket: WebSocket):
        await websocket.accept()
        self.active_connections.append(websocket)
    
    def disconnect(websocket: WebSocket):
        if websocket in self.active_connections:
            self.active_connections.remove(websocket)
```

**2. 总结推送（broadcast_summary）**
```python
async def broadcast_summary(self, summary):
    # 支持两种类型：
    # 1. SummaryResponse对象（Pydantic模型）
    # 2. 字符串类型（最终报告）
    
    if hasattr(summary, 'model_dump'):
        summary_data = summary.model_dump(mode="json")
        await self.broadcast_json({"type": "summary", "content": summary_data})
    else:
        await self.broadcast_json({"type": "summary", "content": summary})
```

**3. JSON广播（broadcast_json）**
```python
async def broadcast_json(self, data: dict):
    # 遍历所有活跃连接，逐个发送
    # 处理发送失败的情况（移除失效连接）
    for connection in list(self.active_connections):
        try:
            await connection.send_json(data)
        except Exception as e:
            self.disconnect(connection)
```

### 4.2 消息路由

**前端 -> 后端：**
- `{"action": "start_query", "query": "..."}`：启动查询
- `{"type": "expand_search", "parent_node_id": "...", "search_type": "...", "search_query": "..."}`：卡片扩展检索

**后端 -> 前端：**
- Graph对象：实时更新图谱状态
- `{"type": "summary", "content": {...}}`：总结报告

## 六、工作流执行流程

### 6.1 完整执行流程

```
1. 用户提交查询
   ↓
2. OrchestratorAgent 接收 UserRequest
   ↓
3. 初始化 ResearchGraph（创建根节点）
   ↓
4. 执行初始搜索（使用用户问题本身）
   ↓
5. ToolExecutorAgent 执行检索工具
   ↓
6. 返回检索结果（RawEvidenceItem列表）
   ↓
7. OrchestratorAgent 处理结果
   - 去重检测
   - 创建/更新节点
   - 构建图谱结构
   ↓
8. 分批发送给 EvaluatorAgent（每批5条）
   ↓
9. EvaluatorAgent 评估每条证据
   ↓
10. 返回评估报告（ItemEvaluation列表）
    ↓
11. OrchestratorAgent 更新节点评估信息
    ↓
12. 检查本轮是否完成（pending_eval_batches == 0）
    ↓
13. 调用 InteractionSummaryAgent 进行本轮总结
    ↓
14. 决策：继续或停止
    - 如果 round_count < max_rounds：执行 _plan_next_move
    - 否则：执行 _finish_task
    ↓
15. _plan_next_move：
    - 构建上下文（历史策略、评估结果、图谱状态）
    - 调用LLM生成新策略
    - 批量分发任务
    - 回到步骤5
    ↓
16. _finish_task：
    - 保存实验结果
    - 保存可视化数据
    - 发送 TaskComplete 给 HypothesisGeneratorAgent
    ↓
17. HypothesisGeneratorAgent 生成最终报告
    ↓
18. 通过WebSocket推送所有结果到前端
```

### 6.2 关键控制机制

**1. 轮次控制**
- `round_count`：当前轮次计数
- `max_rounds`：最大轮次限制（默认7轮）

**2. 批处理控制**
- `pending_tasks`：待处理任务数（工具执行）
- `pending_eval_batches`：待处理评估批次数

**3. 去重机制**
- 基于节点ID的去重（`item.id in self.graph.nodes`）
- 重复节点更新 `hit_count`，不重复评估

**4. 历史策略记录**
- `history_strategies: List[OrchestratorPlan]`：记录所有历史策略
- 用于避免重复策略，提高策略新颖性
